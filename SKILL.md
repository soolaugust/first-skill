# First Principles Thinking

## Overview

将问题分解到最基本的事实，从这些事实重新构建思路。不从类比、惯例或"别人怎么做"出发。

## 强制检查流程

遇到任何分析/设计/调试/决策任务时，必须先走完以下四步：

**Step 1 — 识别假设**
列出你正在依赖的所有假设。包括：
- "通常这样做"
- "上次就是这样解决的"
- "这类问题标准做法是"
- "因为 X 业界都这样"

**Step 2 — 提问"为什么"（至少 3 层）**
对每个假设追问：
- 为什么这是真的？
- 谁证明了这一点？
- 在什么条件下不成立？

**Step 3 — 还原基本事实**
剥离所有假设后，哪些是确定的物理/逻辑/数学事实？

**Step 4 — 从零重建**
从 Step 3 的事实出发，重新推导解决方案。不是"优化现有方案"，而是"如果没有历史包袱，最优解是什么？"

## 禁止的思维模式

| 思维模式 | 为什么有害 | 替换方式 |
|---------|-----------|---------|
| "业界通常这样做" | 惯例可能已经过时或针对不同约束 | 说明该约束在你的场景是否存在 |
| "上次我们用 X 解决了" | 情境不同，方案不可直接复用 | 重新分析当前约束 |
| "参考 Y 框架的做法" | 框架有自己的约束，不一定适合你 | 识别框架解决的核心问题，看你的问题是否一样 |
| "这个领域专家都说应该" | 专家基于其经验的约束，你的约束可能不同 | 理解专家建议背后的前提条件 |
| "感觉应该是这样" | 直觉是压缩过的经验，未必适用 | 展开直觉，找到支撑它的事实 |
| "先做再说，遇到问题再调整" | 从错误方向出发，调整成本指数增长 | 先定义问题边界，再动手 |

## 红旗信号（立即停止重新推导）

出现以下任何一种想法，说明你在走捷径：

- "这明显就是 X 类问题，所以..."
- "照着 [某产品/某论文/某框架] 做就行了"
- "先实现，边做边想"
- "这个领域我熟，不需要从基础推"
- "时间紧，没空分析，直接用常规方案"
- "这是 trivial 的，不需要 first principles"

**时间压力不是跳过分析的理由。** 错误方向上的快速执行比慢速推导浪费更多时间。

## 快速参考

```
问题是什么？（不是"表面现象"，是根本矛盾）
    ↓
有哪些约束？（物理的/逻辑的/资源的）
    ↓
如果约束不存在，理想解是什么？
    ↓
哪些约束是真实的，哪些是"我们一直这样做"？
    ↓
在真实约束下，最接近理想解的方案是什么？
```

## 常见错误

**错误：把"分解"理解为"列子问题"**
分解到基本事实 ≠ 把问题切成小块。是要找到不可再质疑的事实层。

**错误：第一性原理 = 从零发明**
不是要重新发明所有东西。是要理解现有方案的前提条件，确认它们在你的场景中成立。

**错误：分析完了还是选了惯例方案**
从第一性原理推导出来的结论可能和惯例一样。区别是：你知道为什么，而不是因为"大家都这样"。
